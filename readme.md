# Курсовая работа по базам данных

## Описание работы

В рамках курсовой работы были рассмотрены несколько реляционных баз данных в MySQL и PostgreSQL.

Для каждой базы данных в постановке задачи были заранее заданы структура таблиц и тестовые данные. В рамках выполнения курсовой работы основное внимание было уделено разработке SQL-запросов для решения поставленных задач.

Сама цель курсовой работы была в решении поставленных задач.

---

## Цель работы

Целью курсовой работы было закрепление практических навыков проектирования реляционных баз данных и написания SQL-запросов различной сложности.

В процессе выполнения работы были отработаны следующие темы:

- создание связанных таблиц;
- использование первичных и внешних ключей;
- настройка ограничений целостности;
- написание аналитических запросов;
- агрегация и группировка данных;
- объединение данных из нескольких таблиц;
- работа с рекурсивными запросами.

## 1. База данных транспортных средств (01_TransportFacilitiesDB) (MySQL)

### Описание

База данных содержит информацию о транспортных средствах трех типов:

- автомобили;
- мотоциклы;
- велосипеды.

Общая информация о транспортном средстве хранится в таблице `Vehicle`, а характеристики конкретного типа — в специализированных таблицах `Car`, `Motorcycle` и `Bicycle`.

### Реализованные задачи

В рамках данного блока были выполнены запросы для:

- поиска спортивных мотоциклов по заданным критериям мощности и стоимости;
- формирования объединенной выборки автомобилей, мотоциклов и велосипедов с приведением результата к единому набору столбцов.

### Что было отработано

На этом блоке были отработаны:

- соединения таблиц по внешним ключам;
- выборки с разными условиями для разных сущностей;
- объединение результатов через `UNION ALL`;
- сортировка итоговых данных;
- использование `NULL` в случаях, когда часть полей неприменима к определенному типу транспорта.

## 2. База данных автомобилей и гонок (02_AutoRacingDB)(PostgreSQL)

### Описание

База данных посвящена автомобилям, их классам, гонкам и результатам участия.

В структуру базы входят следующие таблицы:

- `Classes` — описание классов автомобилей;
- `Cars` — список автомобилей;
- `Races` — список гонок;
- `Results` — результаты участия автомобилей в гонках.

### Реализованные задачи

Для этой базы были решены задачи, связанные с анализом гоночных результатов:

- определение автомобилей с наилучшей средней позицией в рамках каждого класса;
- поиск автомобиля с наилучшей средней позицией среди всех автомобилей;
- определение лучших классов по средней позиции;
- поиск автомобилей, чья средняя позиция лучше средней позиции по своему классу;
- анализ классов по количеству автомобилей с низкой средней позицией.

### Особенности реализации

Активно использовались:

- многошаговые запросы через `WITH`;
- расчет средних значений;
- подсчет количества гонок;
- выбор лучших значений внутри классов;
- сравнение результатов отдельных автомобилей с агрегированными значениями по классу;
- сортировка по вычисляемым показателям.

### Отдельное замечание по задаче 5

Для пятой задачи были подготовлены два варианта решения.

**Первый вариант** был ориентирован на совпадение с эталонным выводом, приведенным в условии задания. В этом варианте счетчик автомобилей с низкой средней позицией рассчитывается по условию **>= 3.0**, а в итоговый результат выводятся только автомобили со средней позицией больше 3.0.  
**Второй вариант** был построен строго по формулировке задачи. В этом варианте низкой считается только позиция **строго больше 3.0**, выбираются классы с максимальным количеством таких автомобилей, после чего выводятся все автомобили этих классов.

Такое разделение понадобилось из-за того, что логика текстового условия и пример результата интерпретировались по-разному.

## 3. База данных отелей и бронирований (03_HotelBookingDB)(PostGreSQL)

### Описание

База данных моделирует гостиничную систему и включает информацию об отелях, номерах, клиентах и бронированиях.

В структуру базы входят таблицы:

- `Hotel` — список отелей;
- `Room` — номера;
- `Customer` — клиенты;
- `Booking` — бронирования.

### Реализованные задачи

В этом блоке были решены задачи, связанные с анализом клиентской активности:

- поиск клиентов, сделавших более двух бронирований в разных отелях;
- расчет количества бронирований, списка посещенных отелей и средней длительности проживания;
- поиск клиентов, удовлетворяющих одновременно условиям по числу бронирований и сумме затрат;
- определение предпочтительного типа отеля для каждого клиента.

### Особенности реализации

В задачах этого блока использовались:

- подсчет количества бронирований;
- подсчет количества уникальных отелей;
- объединение названий отелей в одно поле;
- категоризация отелей на дешевые, средние и дорогие;
- логика выбора предпочтительного типа отеля клиента на основе посещенных им отелей.

## 4. База данных организационной структуры (04_OrgStructureDB)(PostgreSQL)

### Описание

База данных посвящена организационной структуре компании.

Она включает следующие сущности:

- `Departments` — отделы;
- `Roles` — роли сотрудников;
- `Employees` — сотрудники и их иерархия подчиненности;
- `Projects` — проекты;
- `Tasks` — задачи, назначенные сотрудникам.

### Реализованные задачи

Для этой базы были решены задачи, связанные с иерархией сотрудников и распределением задач:

- поиск всех сотрудников, подчиняющихся Ивану Иванову, включая вложенные уровни подчинения;
- вывод расширенной информации по сотрудникам: отдел, роль, проекты, задачи, количество задач, количество подчиненных;
- поиск сотрудников с ролью менеджера, имеющих подчиненных;
- подсчет количества подчиненных как на одном уровне, так и по всей иерархии.

### Особенности реализации

Главная особенность этого блока — использование рекурсивных запросов.

Для решения задач применялось ключевое слово `WITH RECURSIVE`, которое позволило:

- построить цепочку подчинения;
- получить сотрудников любого уровня вложенности;
- определить количество подчиненных;
- сформировать итоговый отчет по каждому сотруднику с учетом проектов и задач.

## Порядок проверки работы

Так как в работе использовались две СУБД, проверка должна выполняться с учетом конкретного блока:

- `01_TransportFacilitiesDB` — в MySQL;
- `02_AutoRacingDB`, `03_HotelBookingDB`, `04_OrgStructureDB` — в PostgreSQL.

### Общий порядок проверки
Для проверки каждого блока необходимо:

1. создать пустую базу данных;
2. выполнить скрипт создания таблиц (`schema.sql`);
3. выполнить скрипт наполнения тестовыми данными (`data.sql`);
4. последовательно запускать запросы из файла с решениями (`solutions.sql`);
5. сравнивать полученные результаты с ожидаемыми значениями из условий задач.

### Возможные инструменты запуска
- для PostgreSQL: DBeaver, pgAdmin, psql;
- для MySQL: DBeaver, MySQL Workbench, консольный клиент MySQL.

Все решения проверялись на тестовых данных, предоставленных в условиях заданий.
